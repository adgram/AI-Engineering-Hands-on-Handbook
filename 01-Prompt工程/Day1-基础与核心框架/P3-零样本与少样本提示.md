# P3：零样本（Zero-shot）与少样本（Few-shot）提示

## 目标

掌握 Zero-shot 和 Few-shot 两种范式，理解示例数量对输出质量的影响曲线

## 核心概念

### Zero-shot（零样本）
不给任何示例，直接让模型完成任务。

```
System: 将以下句子翻译为英文
User:   今天天气真好
→ 模型依靠预训练知识直接输出
```

### Few-shot（少样本）
使用 `user` + `assistant` 角色，提供 1~N 个示例，让模型学习模式后再回答。

Few-shot 中主要有三种作用：**格式规范化**（教模型输出特定结构，如 JSON/表格）、**风格语气示范**（教模型开口吻、加不加 emoji）、**输出结构示范**（教模型按"结论→理由"等逻辑组织回答）。示例不传授新知识，而是**约束模型的输出空间**。

```
System: 将以下句子翻译为英文
User:   今天天气真好
Assistant: The weather is nice today.
User:   我在吃饭
Assistant: I am eating.
User:   这本书很有趣
→ 模型从示例中推断格式和风格
```

>  `user` + `assistant` 的角色组合，既可以用在 **Few-shot 场景**，也可以用在对话上行文场景（历史提问和模型回复），当这两种情况均出现时，需要根据情况适当组织消息结构，或进行特殊标记，帮助模型区分。

### Few-shot 的变体

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **Static Few-shot** | 示例固定写在代码里，每次请求传相同示例 | 任务格式稳定 |
| **Dynamic Few-shot** | 根据输入动态选择最相似示例 | 示例库大时效果更好 |
| **Many-shot** | 给几十上百个示例 | 复杂任务，效果接近微调 |

> 示例在代码层面有两种传入方式：**标准做法**是通过 `assistant` 角色（`role: "assistant"`）插入，与 `system` / `user` 角色分离，结构清晰且便于程序化动态检索注入；**替代做法**是直接写在 `system` 文本里（如 `"将中文翻译成英文。示例：你好→Hello"`），效果等价但规则与示例混在一起，不推荐。
>
> 下文的实验全部采用标准做法。

## 动手实验

### 实验 1：Zero-shot vs 1-shot vs 3-shot vs 5-shot

在不同的对话中，添加不同数量的示例进行测试。

```python
# 使用特殊格式（【】和 |），模型不通过示例不会自然采用
task = "将产品描述转为营销短文案，格式：【品类】核心卖点1 | 核心卖点2，20字以内"

# 多组测试输入，避免云端缓存命中相同内容
test_cases = [
    "采用 AI 降噪技术的无线蓝牙耳机，续航 30 小时",
    "纯天然有机冷萃咖啡液，0糖0脂，每盒10条装",
]

# 不同品类示例，格式一致：【品类】特性1 | 特性2
examples = [
    {"user": "有机认证的云南咖啡豆，500g 装", "assistant": "【咖啡豆】有机云南 | 500g装"},
    {"user": "纯棉圆领 T 恤，黑白灰三色可选", "assistant": "【T恤】纯棉圆领 | 三色可选"},
    {"user": "不锈钢保温杯，500ml，12小时保温", "assistant": "【保温杯】不锈钢 | 12h保温"},
    {"user": "USB-C 转 HDMI 转换器，4K 60Hz", "assistant": "【转换器】USB-C转HDMI | 4K高清"},
    {"user": "运动跑鞋，网面透气，柔软缓震", "assistant": "【跑鞋】网面透气 | 柔软缓震"},
]
```

**观察**：Zero-shot 不会主动使用 `【】` 和 `|` 格式；1-shot 开始模仿；3-shot 格式趋于稳定；5-shot 输出最规范。多组测试输入可避免云端缓存导致结果雷同。

### 实验 2：不同格式的示例对比

用同一个任务，分别用纯文本/JSON/Markdown 表格格式给示例，看模型是否严格遵循：

```python
# 任务：提取用户信息
test_text = "张三，男，1995年出生，北京大学计算机系毕业，现在腾讯做后端开发，擅长 Go 和 Python"

# 格式A：纯文本
few_shot_text = [
    {"role": "system", "content": "提取用户信息"},
    {"role": "user", "content": "李四，女，28岁，本科，设计师"},
    {"role": "assistant", "content": "姓名：李四，性别：女，年龄：28，学历：本科，职业：设计师"},
    {"role": "user", "content": test_text}
]

# 格式B：JSON
few_shot_json = [
    {"role": "system", "content": "提取用户信息，输出 JSON"},
    {"role": "user", "content": "李四，女，28岁，本科，设计师"},
    {"role": "assistant", "content": '{"name": "李四", "gender": "女", "age": 28, "education": "本科", "job": "设计师"}'},
    {"role": "user", "content": test_text}
]

# 格式C：Markdown表格
few_shot_table = [
    {"role": "system", "content": "提取用户信息，输出 Markdown 表格"},
    {"role": "user", "content": "李四，女，28岁，本科，设计师"},
    {"role": "assistant", "content": "| 姓名 | 性别 | 年龄 | 学历 | 职业 |\n|------|------|------|------|------|\n| 李四 | 女 | 28 | 本科 | 设计师 |"},
    {"role": "user", "content": test_text}
]
```

**观察**：模型会严格复制示例的输出格式。这是 Few-shot 最核心的用途——**格式示范**。

### 实验 3：示例顺序的影响

模型对靠近问题位置的示例关注度更高（**最近优先**效应）。本例用 emoji 风格来直观演示——有序时所有回复带 emoji，乱序时将不带 emoji 的示例放在最后，看模型是否会被"带偏"。

```python
# 有序：3 个示例都带 emoji，格式一致
examples_ordered = [
    {"user": "今天天气真好", "assistant": "☀️ 适合出去玩"},
    {"user": "工作好忙啊", "assistant": "💼 加油打工人"},
    {"user": "周末到了", "assistant": "🎉 好好休息"},
]

# 乱序：最后一个示例故意不放 emoji
examples_shuffled = [
    {"user": "周末到了", "assistant": "🎉 好好休息"},
    {"user": "今天天气真好", "assistant": "☀️ 适合出去玩"},
    {"user": "工作好忙啊", "assistant": "工作再忙也要按时吃饭"},
]
```

**观察**：有序组输出通常带 emoji（模仿前 3 个示例）；乱序组输出可能不带 emoji（受最后一个无 emoji 示例影响更大）。这就是**最近优先**效应——越靠近问题位置的示例影响力越大。

### 实验 4：assistant 角色格式示范

演示在 `assistant` 角色中写入示例消息，让模型学会输出格式——不加示例模型自由发挥，加示例后学会结构化输出：

```python
# 不加示例：模型自由发挥
messages_no_example = [
    {"role": "system", "content": "你是一个会议纪要助手"},
    {"role": "user", "content": """整理以下会议记录：
参会人：张总、李工、王经理
内容：讨论了 Q3 预算审批，决定追加 20 万用于服务器升级。
李工提出前端重构方案，预计 8 月底完成。
王经理建议下周一前各团队提交进度报告。"""}
]

# 加示例：让模型学会结构化格式
messages_with_example = [
    {"role": "system", "content": "你是一个会议纪要助手"},
    {"role": "user", "content": """整理以下会议记录：
参会人：王总、刘工
内容：讨论了登录页改版方案，决定采用方案 B。
刘工建议周四前出原型图。"""},
    {"role": "assistant", "content": """## 议题：登录页改版
- **决定**：采用方案 B
- **待办**：刘工周四前出原型图"""},
    {"role": "user", "content": """整理以下会议记录：
参会人：张总、李工、王经理
内容：讨论了 Q3 预算审批，决定追加 20 万用于服务器升级。
李工提出前端重构方案，预计 8 月底完成。
王经理建议下周一前各团队提交进度报告。"""}
]
```

**观察**：不加示例时模型输出大段描述性文字；加上 `assistant` 示例后，模型自动学会"议题 → 决定 → 待办"的结构化格式。这正是 Few-shot 的本质——通过 `assistant` 角色的示例消息来示范目标输出格式。

## 事实性与偏差

示例的组成与顺序，对模型的判断也存在影响。在制作示例时应有合适的数据级。

### 示例分布偏差

Few-shot 中示例的比例会影响模型的输出倾向：

```
# 偏差示例：3 个正面 + 1 个负面
示例1：好评 → 正面 
示例2：好评 → 正面
示例3：好评 → 正面
示例4：差评 → 负面
测试输入：中性评论 → 模型倾向于输出"正面"

# 修正：均匀分布
示例1：好评 → 正面
示例2：差评 → 负面
示例3：好评 → 正面
示例4：差评 → 负面
测试输入：中性评论 → 更可能输出"中性"
```

> 原则：示例中每个类别的比例尽量均等，避免无意中引入分布偏差。

### 顺序偏差

同一组示例，顺序不同可能导致不同结果：

```
# 顺序 A：正面示例在前
[好评→正面, 好评→正面, 差评→负面, 差评→负面]
→ 模型更偏好正面

# 顺序 B：负面示例在前
[差评→负面, 差评→负面, 好评→正面, 好评→正面]
→ 模型更偏好负面
```

> 应对：多次运行取平均，或随机打乱示例顺序后固定使用。

## 扩展：文本转换（Transformation）

提示词不仅用于问答，还可以完成各种文本格式的转换任务：

| 任务 | 示例 Prompt | 用途 |
|------|------------|------|
| **翻译** | "将以下中文翻译成英语" | 跨语言沟通 |
| **语气转换** | "将以下非正式消息转换为正式商务邮件" | 适配不同受众 |
| **格式转换** | "将 JSON 转换为 HTML 表格" | 数据展示 |
| **拼写/语法检查** | "校对并修正以下文本" | 质量保障 |

```python
# 语气转换示例：非正式 → 正式
messages = [
    {"role": "system", "content": "你是一个专业商务写作助手"},
    {"role": "user", "content": "将以下非正式消息转换为正式的商务邮件：\n'嗨，我是小王，你看看这个新产品的介绍，挺不错的。'"}
]
```

> 提示：转换类任务的关键是明确描述"输入格式"和"输出格式"，模型会自动完成映射。

## 关键洞察
1. **约束输出空间**：在 Few-shot 场景下，示例的本质不是传授知识，而是告诉模型"输出应该长什么样"——限定范围、格式、风格（多轮对话中的 assistant 历史消息承载的是事实性上下文，不属于示例范畴）
2. **收益递减**：通常 3~5 个示例就够，再多效果提升很小但 Token 成本线性增加
3. **示例质量 > 数量**：一个高质量示例胜过五个模糊示例
4. **Dynamic Few-shot** 是生产环境常用策略：从示例库中检索最相似的 N 个
5. **Zero-shot 的优势**：零成本，适合简单、明确的任务
