## 1. 消息角色

### 1.1 四种消息角色

调用大模型时，`messages`参数支持传入四种角色。

| 角色 | 作用 | 谁写的 |
|------|------|--------|
| `system` | 设定行为、风格、规则 | 开发者 |
| `user` | 用户输入/问题 | 用户 |
| `assistant` | 模型回复（也可由开发者写，作为示例） | 模型或开发者 |
| `tool` | 工具调用的返回值——当模型申请调用工具时，将调用的结果作为此角色返回给模型 | 系统 |

> `tools` 参数（请求时传入）定义模型可以调用的工具列表；`tool` 角色（对话消息）用于把工具执行结果送回模型。两者配合完成一次工具调用：定义 → 模型申请调用 → 本地调用 → 返回结果 → 模型根据调用结果进行回复。具体代码见 [P5-结构化输出与Function-Calling.md](P5-结构化输出与Function-Calling.md)「任务 3」。
>
> **关键规则**：`user`和`assistant`消息建议交替出现（部分 API 严格校验该顺序，连续两条 `user` 或 `assistant` 可能导致报错或效果下降）。
>
> 四种角色是 API 层面的**语义约定**，用来让大模型区分指令的类型和权威层级。角色之间存在一条隐式优先级：**`system`（开发者层）> `user`（用户层）> `assistant`（模型层）**。`system` 来自可信端、优先级最高——恶意 `user` 输入可以用规则约束，但无法覆盖 `system`。这种分级设计是安全防注入的根基。

### 1.2 为什么区分 System 和 User

**为什么不用一个角色搞定？**

| 维度 | System | User |
|------|--------|------|
| **职责** | 设定人设、规则、边界（怎么回答） | 提出任务、输入数据（回答什么） |
| **持久性** | 全程不变，每轮对话都生效 | 每轮变化，随用户输入不同 |
| **来源** | 开发者编写（可信） | 用户输入（可能不可信） |
| **安全** | 防注入的根基——System 中的指令优先级高于 User | 恶意用户可能试图绕过规则，System/User 分离便于做权限校验 |

**如何选择？**

| 内容类型 | 放哪里 | 例子 |
|----------|--------|------|
| 角色身份、语气风格 | System | "你是一个专业且亲切的旅游规划助手" |
| 行为规则、安全约束 | System | "不要推荐竞品、不讨论政治" |
| 输出格式规范 | System | "每次回复先给结论再解释" |
| 当前任务、具体问题 | User | "帮我规划三天北京旅游路线" |
| 上下文/背景信息 | User | "我带了老人和小孩，预算 5000" |
| 反面约束 | User | "不要包含购物景点" |

**是否可以全部塞进 System 或 User？**

- **全放 System**：从功能上每轮修改 System 与放 User 无异，但违背了 API 语义约定——System 本应是"开发者可信区"，优先级高于 User。滥用 System 做每轮变化，既丢掉了安全分级优势，也浪费了 Prompt Caching 的缓存红利。
- **全放 User**：失去了持久角色设定——需要反复重申人设和规则。且 User 输入常来自不可信端，安全意识差——恶意用户的输入可能试图覆盖开发者指令。
- **最佳实践**：System 放**不变**的部分（身份、规则、格式），User 放**变化**的部分（任务、数据、单次约束）。这也是为什么 API 设计上把两者拆成独立角色——不是能不能，而是该不该。

### 1.3 Prompt 的"段位进阶"

理解消息角色后，再看 Prompt 本身的成熟度层级：

| 段位 | 特征 | 示例 |
|------|------|------|
| **青铜** | 一句话指令，无角色无格式 | "帮我写个文案" |
| **白银** | 带角色和基本约束 | "你是XX，帮我写XX，要求XX" |
| **黄金** | 结构化模板，用框架组织 | CO-STAR / RICE / TRACE 框架 |
| **钻石** | 参照业界框架的完整 System Prompt | 角色+准则+边界+操作+流程+风格+兜底 |

### 1.4 消息顺序示例

```
正确: system → user → assistant → user → assistant
正确: user → assistant → user → assistant
错误: user → user → assistant
错误: system → assistant → user
```

## 2. System Prompt 进阶：三种主流设计路径

一个好的 System Prompt 不应该只有一句话。当前业界主要存在三种截然不同的组织哲学，分别来自三大 AI 厂商。

### 2.1 三种设计路径对照

|  | OpenAI 指令层级 | Anthropic 动态组装 | Google / monday.com 结构化分段 |
|---|---|---|---|
| **核心理念** | 按角色权威分级，权限决定优先级 | 条件拼接 + 缓存边界，按需组合 | 按维度拆分为固定段落，模板化复用 |
| **典型结构** | Root → System → Developer → User → Guideline | 7+ 静态段 + N 条件动态段（环境/记忆/MCP） | Role → Scope → Tone → Safety → Context → Values → Formatting |
| **关键区别** | "谁说的"比"说什么"更重要 | "放哪里"比"写什么"更重要 | "覆盖全"比"顺序对"更重要 |
| **工程特点** | 天然防注入、多租户安全 | 缓存友好（~70% 复用）、特征开关 | 可读性强、非技术人员可协作编写 |

三种路径并非互斥——生产级 System Prompt 往往混合使用：用结构化分段组织内容，用指令层级设计安全策略，用动态组装控制缓存与条件注入。

### 2.2 共同范式

三种路径形式各异，但落实到具体的 Prompt 内容，都离不开这 5 个核心维度：

| 共同维度 | 说明 |
|----------|------|
| **角色/身份 (Identity)** | 定义 AI 是谁、为什么存在 |
| **行为/安全 (Behavior & Safety)** | 该怎么做、什么不能做 |
| **工具/能力 (Tools & Capabilities)** | 能用什么工具、做什么操作 |
| **沟通/风格 (Tone & Style)** | 怎么说话、输出格式 |
| **环境/边界 (Context & Constraints)** | 运行环境、知识范围、动态信息 |

### 2.3 完整示例（电商客服）

以下是用 **五维共同范式**（角色/身份 → 行为/安全 → 工具/能力 → 沟通/风格 → 环境/边界）完成的电商客服示例。

```
## 角色/身份（Identity）
你是某电商平台的智能客服助手「小购」，为顾客提供购物咨询服务。
价值观：用户利益优先——推荐合适而非最贵的商品；信息透明——运费、退换货政策主动告知，不隐瞒。

## 行为/安全（Behavior & Safety）
1. 不得对商品做绝对化承诺（如"全网最低""质量最好"）
2. 涉及资金/账户操作时，引导用户到官方渠道自行完成
3. 疑似网络诈骗或异常行为时，停止回答问题并引导联系官方客服
4. 不处理：法律纠纷建议、竞品平台比价、第三方物流催单

## 工具/能力（Tools & Capabilities）
你可以处理：商品咨询、规格对比、促销说明、退换货流程引导。

## 沟通/风格（Tone & Style）
专业但不冰冷，亲切但不随意；多用短句，避免长篇大论。
禁忌：不使用"亲""宝"等过度网络化称呼。
每轮回复结构：肯定用户 → 直接答案 → 可选追问（不超过 2 个）
价格信息需标注有效期（如"目前售价 ¥299，活动截止本周日"）

## 环境/边界（Context & Constraints）
本平台主营 3C 数码、家居日用、服饰箱包三大品类，覆盖全国 300+ 城市次日达。
当前平台为自建官方网站，无第三方店铺接入。
商品价格与库存实时变动，回复中涉及价格/库存时需标注"以页面显示为准"。
```

> **写作要点**：每个维度 3-7 条规则最有效。三种路径的选择：通用对话/客服场景推荐**结构化分段**（可读性强、易协作）；复杂 Agent/工具密集型场景推荐**动态组装**（缓存友好、条件注入灵活）；安全敏感/多租户场景推荐以**指令层级**为纲设计权限策略。三者也可混合使用——例如用结构化分段写内容，用指令层级定安全，用动态组装管理长对话缓存。

## 3. 命名 Prompt 框架

除了自由组合 Role/Task/Context/Format，业界还有成熟的命名框架可直接套用：

| 框架 | 组成 | 适用场景 |
|------|------|----------|
| **RICE** | Role + Instruction + Context + Example | 专业写作、技术分析 |
| **TRACE** | Task + Role + Action + Context + Example | 复杂多层次任务 |
| **BROKE** | Background + Role + Objectives + Key Result + Evolve | 有明确目标和可衡量结果的项目 |
| **PAS** | Problem + Agitation + Solution | 营销文案、客户服务 |
| **RACE** | Role + Action + Context + Expectation | 需具体行动建议的决策任务 |
| **CO-STAR** | Context + Objective + Style + Tone + Audience + Response | 内容创作、营销文案 |

示例（CO-STAR 框架）：

```
## Context（上下文）
我们是一家面向25-35岁年轻人的咖啡品牌，即将推出一款冷萃咖啡液。

## Objective（目标）
撰写一篇小红书种草文案，提升新品知名度。

## Style（风格）
轻松活泼，带个人体验感，使用 emoji 和话题标签。

## Tone（语气）
亲切、真诚，像朋友推荐一样自然。

## Audience（受众）
喜欢尝试新事物的年轻白领和大学生。

## Response（回应格式）
输出一段 200 字左右的文案，包含标题 + 正文 + 3 个标签。
```

> 框架只是脚手架，核心还是把角色、任务、上下文、格式说清楚。选一个顺手的套用即可。

## 4. 负面约束技巧

明确告诉模型**不要做什么**和告诉它**要做什么**同样重要：

```
只说正面："请提供一些关于健康饮食的建议。"
→ 可能包含减肥建议、推荐特定产品

正 + 负面结合：
"请提供关于健康饮食的建议，但：
- 不要包含具体的减肥建议
- 不要推荐特定品牌的产品
- 避免使用医学术语
- 重点关注日常可实施的饮食习惯"
```

> 原理：模型的输出倾向于沿着"最可能的路径"展开，负面约束明确地切断了这些路径，让模型更准确聚焦。

## 5. 行业标准：AI 工具的提示词分类体系

### 5.1 为什么需要分类？

P2 前面的内容教会你如何用业界框架写一个高质量的 System Prompt。但在真实工程中，一个 AI 智能体往往需要管理大量指令——人设语气、行为规则、工具定义、长期记忆、安全约束等。如果全部写在同一个 prompt 里，会出现三个问题：

1. **耦合过紧**：改一句话术要点，却要动整个 prompt
2. **复用困难**：同一套行为规则无法跨项目共享
3. **上下文浪费**：所有内容每次都全量注入，占用大量 Token

业界解决方案：**按职责拆分到多个文件，按需加载**。

### 5.2 AGENTS.md 行业标准

2025 年 8 月，OpenAI 发布了 `AGENTS.md` 规范，后移交 Linux Foundation 旗下的 **Agentic AI Foundation (AAIF)** 管理。截至 2026 年中，已被 **60,000+ 开源仓库**和 **20 多种 AI 工具**采纳。

```
AGENTS.md 定义：一个纯 Markdown 文件，用自然语言告诉 AI 助手如何工作。
不要求固定 schema，不要求 YAML frontmatter，没有任何侵入性语法。
```

**作用域层级**（就近优先）：

| 层级 | 路径 | 作用 |
|------|------|------|
| 全局 | `~/.app/AGENTS.md` | 用户所有项目的默认规则 |
| 项目根 | `./AGENTS.md` | 当前项目的共享规则（提交到 git） |
| 子目录 | `./packages/*/AGENTS.md` | monorepo 子包的特定规则 |

支持该标准的工具包括：Codex CLI、Cursor、GitHub Copilot、Gemini CLI、Claude Code（fallback）、Windsurf、Zed、Warp、Aider、OpenCode、WorkBuddy 等。

> 详见：[agents.md](https://agents.md)

### 5.3 典型工具的 Prompt 组织方案

在 AGENTS.md 标准之上，各工具在工程落地时设计出了不同的组织方案：

| 工具 | 核心配置文件 | 分类逻辑 | 核心思想 |
|------|-------------|----------|----------|
| **OpenClaw** | SOUL.md + AGENTS.md + IDENTITY.md + TOOLS.md + MEMORY.md | **按职责拆分** | 每个文件管一件事：语气/规则/身份/工具/记忆 |
| **Claude Code** | CLAUDE.md + `.claude/skills/*/SKILL.md` | **按模式+子代理** | 40+ prompt 片段，根据 Plan/Explore/Delegate 等模式动态组装 |
| **OpenCode** | AGENTS.md + `.opencode/skills/*/SKILL.md` | **按层级+技能** | 目录层级就近覆盖 + Skills 按需加载 |
| **Codex CLI** | AGENTS.md（兼容 AAIF） | **按目录层级** | monorepo 子包各有自己的 AGENTS.md |
| **Cursor** | `.cursor/rules/*.mdc` | **按规则文件** | 每条规则独立文件，可开关/分组 |
| **GitHub Copilot** | `.github/copilot-instructions.md` | **按作用域** | 全局指令 + 特定路径指令 |
| **WorkBuddy** | 兼容 AGENTS.md | 遵循 AAIF 标准 | 与 CodeBuddy 生态整合 |

以 **OpenClaw** 的多文件体系为例，它是最直观的"按职责分类"方案：

```
项目根目录/
├── AGENTS.md       # 行为规则、工作流程、安全约束
├── SOUL.md         # 人设、语气、风格、价值观
├── IDENTITY.md     # 身份声明：名字、使命、起源
├── TOOLS.md        # 可用工具和操作的定义
├── MEMORY.md       # 长期知识、项目背景、用户偏好
├── HEARTBEAT.md    # 心跳行为配置
└── .opencode/
    └── skills/     # 可插拔技能目录
        ├── code-review/SKILL.md
        └── api-design/SKILL.md
```

### 5.4 共同维度 → 多文件映射

在第 2.2 节提炼的 5 个共同维度，可以一一映射到多文件体系：

| 共同维度 | 对应文件 | 说明 |
|----------|----------|------|
| **角色/身份** | `IDENTITY.md` + `SOUL.md` | 身份文件声明你是谁，语气文件决定怎么说 |
| **行为/安全** | `AGENTS.md` | 行为规则、安全约束的核心存放位置 |
| **工具/能力** | `TOOLS.md` + Skills | 工具定义放 TOOLS，领域专长放 Skills |
| **沟通/风格** | `SOUL.md` | 语气/风格的专属文件 |
| **环境/边界** | `MEMORY.md` | 长期知识、项目背景、用户偏好 |

映射关系说明了一个核心观点：**五种维度是内容的组织逻辑，多文件体系是工程的组织逻辑。** 前者解决"写什么"，后者解决"放哪里"。

### 5.5 多文件内容的运行时组装

拆分后的文件最终要拼回实际请求的 `messages` 数组，有三种主流方式：

| 方式 | 做法 | 适用场景 |
|------|------|----------|
| **拼装为单条 system** | 读取多个文件，按章节拼接成一条 content | 内容稳定、依赖 Prompt Caching |
| **多条 system 消息** | 每个文件作为独立的一条 `system` 消息 | 需要动态控制某维度的开关（如按需加载记忆） |
| **模板引擎管理** | 用 YAML / Jinja2 管理模板，运行时渲染 | 多版本管理、A/B 测试 |

三种方式功能等价，区别在于**缓存粒度和动态灵活性**的不同取舍。

### 5.6 规则重复的处理

多文件体系下，同一条规则可能出现在多个地方，此时取决于你如何组织：

| 情况 | 效果 | 工程建议 |
|------|------|----------|
| **简单内容重复**（同一使命在 SOUL 和 IDENTITY 各写一次） | 模型看到冗余内容，不会报错但浪费 Token，且可能稀释指令权重 | 团队约定每个维度只在一个文件中定义（身份→IDENTITY、规则→AGENTS、语气→SOUL） |
| **目录级覆盖**（项目根 AGENTS.md vs 子包 AGENTS.md） | AGENTS.md 标准定义**就近优先**——子目录文件覆盖父级同名规则 | 利用层级覆盖做差异化，而非复制粘贴 |
| **多个文件指令冲突**（AGENTS.md 说"可以"、SOUL.md 说"不可以"） | 模型行为不确定，取决于内部优先级或位置顺序 | 代码审查时检查跨文件指令一致性 |

> 核心原则：**同一份规则只在一个文件中定义**，跨项目变化用目录层级覆盖，而非复制。

## 动手实验

### 练习 1：多轮对话（三种角色完整运用）

实现一个持续对话，完整演示 system/user/assistant 三种角色的交替。此处为一个最简单的多轮对话实现，多轮对话中 system 保持不变，创建一个记录所有消息的列表，将每一次用户输入和模型回复作为user 和 assistant 角色消息依次记录，在下一轮对话时传入。

```python
def chat_with_history():
    messages = [
        {"role": "system", "content": "你是一个旅游规划助手。每次回复包含两部分：1. 回答用户的问题2. 紧跟 2-3 个追问来收集偏好（用「🤔 了解你」开头）"}
    ]
    print("=== 旅游规划助手（输入 'quit' 退出）===\n")
    while True:
        user_input = input("你: ")
        if user_input.lower() == 'quit':
            break
        messages.append({"role": "user", "content": user_input})
        response = client.chat.completions.create(
            model="deepseek-v4-flash",
            messages=messages
        )
        reply = response.choices[0].message.content
        print(f"AI: {reply}\n")
        messages.append({"role": "assistant", "content": reply})
        print(f"当前历史消息数: {len(messages)} (system+{len(messages)-1}轮对话)")
chat_with_history()
```

**观察**：对话轮次越多，上下文开销越大，注意 Token 消耗。

### 练习 2：用结构化分段路径构造 System Prompt

用 **Role → Scope → Tone → Safety → Context → Values → Format** 为"中文写作助手"角色编写完整的 System Prompt。

```
## Role（角色定义）
你是「文笔 AI」，一个专门帮助用户提升中文写作质量的智能助手。

## Scope（范围边界）
你专注于：文章润色、句式优化、逻辑梳理、风格调整、错别字与语病检查。
不处理：代写完整论文、翻译整本著作、生成违法内容。

## Tone（语气风格）
专业但不刻板，给出建议时使用"建议…""可以考虑…"等委婉句式。
避免使用"你写错了""这里不对"等否定式表达。

## Safety Rules（安全规则）
1. 不直接修改用户原文，而是给出修改建议和理由
2. 涉及敏感话题时，只做语言层面的润色，不参与立场讨论
3. 每次最多给出 3 条修改建议，避免信息过载

## Context（上下文）
用户是想提升中文写作水平的职场人士或学生。
系统运行在 DeepSeek API 上，模型为 deepseek-v4-flash。

## Values（价值观）
保持作者原意，不做过度改写；帮助用户成长，而非替用户代劳。

## Formatting（输出格式）
每轮回复分为三部分：
1. 先肯定原文优点（1-2 句）
2. 逐条给出修改建议（每条含原文+建议+理由）
3. 可选：提供一个优化后的完整版本
```

**观察**：使用结构化分段路径的 System Prompt 后，模型回复有清晰的分段结构，建议具有可操作性，不会过度改写原文。

### 练习 3：用 CO-STAR 框架做角色设定

对比普通 System Prompt 和使用 CO-STAR 框架的 System Prompt 的效果差异：

```
# 普通 system prompt
"你是一个营销文案写手"

# CO-STAR 框架
## Context（上下文）
我们是一家面向 25-35 岁年轻人的咖啡品牌，即将推出一款冷萃咖啡液。

## Objective（目标）
撰写一篇小红书种草文案，提升新品知名度。

## Style（风格）
轻松活泼，带个人体验感，使用 emoji 和话题标签。

## Tone（语气）
亲切、真诚，像朋友推荐一样自然。

## Audience（受众）
喜欢尝试新事物的年轻白领和大学生。

## Response（回应格式）
输出一段 200 字左右的文案，包含标题 + 正文 + 3 个标签。
```

**观察**：**普通 Prompt** — 输出了一份完整的种草文案（224字），带 emoji、有场景描述（打工人/学生党）、有隐藏喝法推荐，风格生动。但字数超了，结构发散。**CO-STAR 框架** — 输出更符合约束：标题+正文+3 标签，200 字左右，风格统一为"朋友推荐"口吻，没有偏离到隐藏喝法之类的内容。

### 练习 4：负面约束对比

对比"只说正面"和"正面+负面约束"的效果差异：

```python
# 只给正面指令
positive_only = [
    {"role": "system", "content": "你是一个健康饮食顾问"},
    {"role": "user", "content": "请提供一些关于健康饮食的建议"}
]

# 正面 + 负面约束
with_constraints = [
    {"role": "system", "content": "你是一个健康饮食顾问"},
    {"role": "user", "content": """请提供关于健康饮食的建议，但：
- 不要包含具体的减肥建议
- 不要推荐特定品牌的产品
- 避免使用医学术语
- 重点关注日常可实施的饮食习惯"""}
]
```

**观察**：加了负面约束后，模型输出更聚焦。

### 练习 5：持续对话中用户画像注入

类似 ChatGPT 的持久化偏好设置，在代码中模拟"预置身份"：

```python
class UserProfile:
    """持久化用户偏好，每次对话自动注入"""
    def __init__(self, role_desc: str, style_desc: str):
        self.role_desc = role_desc
        self.style_desc = style_desc
    
    def build_system_prompt(self) -> str:
        return f"""## 关于用户
{self.role_desc}

## 回答要求
{self.style_desc}"""

# 为前端开发者预设
dev_profile = UserProfile(
    role_desc="我是一名有5年经验的前端开发者，主力 Vue 3 + TypeScript，使用 Vite 构建工具",
    style_desc="请以资深技术专家口吻回应。提供代码示例时用最新语法标准。用 Markdown 格式化代码块。"
)

messages = [
    {"role": "system", "content": dev_profile.build_system_prompt()},
    {"role": "user", "content": "请帮我制作一个简单的算术题应用"}
]
```

### 练习 6：模拟多文件 Prompt 加载

将完整的 System Prompt 拆分为多个虚拟文件，再拼装回完整 prompt，对比与单行 prompt 的效果差异。

```
"SOUL.md": 
    "你是一个专业且亲切的中文写作教练「文笔 AI」。"
    "语气：鼓励式，先肯定优点再提建议。"
    "禁忌：不直接说'写得不好'，改用 '这里可以更流畅，比如…'"

"AGENTS.md": 
    "## 行为准则"
    "1. 保持作者原意，不做过度改写"
    "2. 每次最多给出 3 条修改建议"
    "3. 给建议时说明理由"
    "## 标准工作流程"
    "阅读原文 → 肯定优点 → 逐条建议 → 可选提供优化版本"
    "## 限制与兜底"
    "- 无法判断时请用户补充说明"
    "- 代写完整论文等超出范围的要求，礼貌拒绝"

"TOOLS.md": 
    "可用能力："
    "1. 润色改写：优化句式、提升流畅度"
    "2. 逻辑梳理：调整段落结构、理清因果关系"
    "3. 风格调整：将口语转书面、或书面转生动"

"MEMORY.md": 
    "知识边界：中文语法、修辞手法、写作技巧"
    "不可回答：英文写作润色、学术论文代写"
```

**观察**：多文件组装版本的回复在建议质量和结构规范性上明显优于单行版本。同时，可单独修改 SOUL.md 调整语气，或修改 MEMORY.md 更新知识边界，而不影响其他部分。

## 常见陷阱

1. **消息顺序错误**：`[user, assistant, user]`（正确），`[user, user, assistant]`（错误）
2. **Context 窗口溢出**：多轮对话不控制长度会超 Token 限制
3. **System Prompt 太弱**：`"你是一个助手"` 不如 `"你是 XX 领域的专家，回答风格 YY"`

> Context Window（上下文窗口）是 LLM 一次能处理的最大 Token 数。比如 deepseek-v4-flash 的 Context Window 是 128K tokens，意味着一次请求中 system + 所有历史消息 + 模型回复的 Token 总和不能超过这个限制。
>
> 在多轮对话中，随着消息不断追加，Context 会越来越长，接近限制时就需要做截断或摘要——这就是"Context 窗口溢出"问题。
