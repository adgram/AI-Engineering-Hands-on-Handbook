# Prompt工程 | P9：拓展阅读 — Prompt 工程进阶知识

本节介绍的六个知识领域（思维链、模板化、陷阱与调试、自动优化、FC 进阶、Harness 工程）是 Prompt 工程中重要的进阶内容。它们与 P1-P7 的核心技能共同构成了完整的 Prompt 工程知识体系，也是后续学习 RAG 和 Agent 的重要铺垫。

## 一、思维链（Chain-of-Thought, CoT）

### 什么是思维链

思维链的核心思想是让模型在给出最终答案前先展示推理过程。对于复杂推理任务（数学题、逻辑推理、多步决策），直接要求模型输出答案容易导致"跳步"错误，而让模型一步步推理可以显著提升准确率。

### Zero-shot CoT

2022 年 Kojima 等人提出：在 Prompt 末尾加上"让我们一步步思考"就可以触发模型的逐步推理。这个技巧对非推理模型效果显著，但对于 DeepSeek V4 这样内置推理能力的模型几乎无额外效果，因为模型内部已经在做分步推理。

### Few-shot CoT

通过在示例中包含推理过程（在 assistant 角色中给出完整的推导步骤），让模型模仿相同的推理风格。示例中推理步骤越清晰，模型复现越准确。适用于需要特定推理风格或复杂逻辑的任务。

### 自我修正（Self-Reflection）

先生成初始版本，再让模型以"审查者"角色审视自己的输出并优化。模型在审查者角色下会比创作者角色更严格地发现问题，适用于代码审查、文案修改、逻辑验证等场景。

### 适用建议

对于内置推理能力的模型，不需要手动加简单的 CoT 提示词。CoT 的收益与长度呈倒 U 型曲线——简单问题花大量 tokens 推理并不能提高准确率，纯属浪费。直接回答、正常推理、过度推理三者都可能答对，但 token 消耗差数倍到数十倍。简单题用直接回答节省成本，复杂题再用 CoT。

## 二、Prompt 模板化

### 为什么需要模板

当 Prompt 超过 20 行、有多个变量、多种角色时，硬编码方式（f-string 直接写在业务代码中）会导致代码耦合严重、难以维护。模板化将 Prompt 的定义与调用分离，提升复用性和可维护性。

### 三种主流方案

**方案一：Python f-string / str.format**。最轻量的方式，适合个人项目或小团队。将模板定义为静态方法集中管理，调用时传入参数渲染。优点是无额外依赖，缺点是缺乏动态逻辑。

**方案二：Jinja2 模板**。支持循环、判断、过滤器等逻辑处理，适合复杂条件分支。例如根据场景动态拼接角色设定、规则、示例等。优点是灵活强大，缺点是需要引入第三方库。

**方案三：YAML 配置管理**。将模板与代码完全分离，存储在 YAML 文件中，非技术人员可编辑。配合简单的 PromptManager 类，可以实现模板注册、渲染、持久化的完整管理。适合多人协作和多版本场景。

### 工程建议

先用 Python 原生方式熟练掌握模板化思路，再引入框架。过早引入框架反而会掩盖底层原理。模板化的本质是"把 Prompt 当成数据来管理"。

## 三、常见陷阱与调试最佳实践

### 六大常见陷阱

**指令冲突**：同时在 System Prompt 中要求"创意"和"简洁"是矛盾指令，模型无法同时满足。指令要一致，避免内耗；用具体数字替代模糊描述。

**上下文过载**：长上下文中模型倾向于记住开头和结尾的信息，中间部分容易被忽略（Lost in the Middle 现象）。重要信息应放在开头和结尾，参考信息放中间。

**过度指令**：过多的强调词（"必须"、"一定要"、"绝对不要"）反而降低模型注意力。正面指令加明确负面指令各一条即可。

**隐含假设**：模型不知道你的代码背景、项目环境或前置条件。提供足够背景信息，不要假设模型"应该知道"。

**模型幻觉**：模型在知识截止日期后的事件、训练数据中少见的知识面前，会自信地编造答案。应对方式包括提供参考信息、使用 RAG、要求模型标注不确定程度。

**缺乏兜底**：模型无法回答时编造答案而非承认不知道。应在 Prompt 中明确告诉模型"不知道就说不知道"。

### 调试标准流程

明确任务目标 → 写第一个版本 → 测试 5-10 个不同用例 → 分析错误模式（格式错误、内容错误、遗漏信息）→ 针对性修改 Prompt → 重复测试直到满意 → 固定为模板。

### 五维评估框架

评估 Prompt 质量可以从五个维度入手：准确率（测试集预期输出匹配率）、完整性（是否遗漏需求要点）、一致性（多次运行结果是否稳定）、安全性（对注入攻击的抵抗能力）、体验（输出格式、语气、可读性）。每次修改 Prompt 前先跑一次基线，改完再跑一次，确保变化是有价值的。

## 四、Prompt 自动优化与循环工程

### 从手动到自动

传统的手动迭代优化依赖人工分析错误模式并修改 Prompt，每轮都靠人盯着。自动优化的核心思想是"反思式进化"——让 AI 从自己的错误中学习，形成自动闭环。

### 三种自动优化范式

**GEPA（Generative Evolutionary Prompt Adapter）**：采用 Generator、Evaluator、Optimizer 三个角色分工协作。Generator 基于当前 Prompt 和错误分析生成候选变体，Evaluator 用测试集评分，Optimizer 分析失败模式给出改进方向。多角色分工可以避免自评偏见，最鲁棒。

**Loop Engineering（循环工程）**：在 GEPA 基础上增加了工程化约束——增量修改（每轮只改 1-2 个 section）、棘轮机制（只保留分数不降的版本）、三重隔离（生成/评分/迭代环境相互隔离防止作弊）。工程化最完整，适合生产环境。

**SPO（Self-Reflective Prompt Optimization）**：只用单一模型、单一角色完成全部流程，让模型自我分析、自我批判、自我重写、自我评估。实现成本最低，但自评虚高风险最严重。适合 1-2 轮快速验证。

### 关键工程原则

**棘轮机制**：每次迭代后只保留分数不降的版本，拒绝任何回退，确保优化方向单调递增。

**三重隔离**：修改 Prompt 的 AI 不能给自己打分，否则会产生严重的自评虚高。生成环境、评分环境和迭代 Agent 应互相隔离。

**增量修改**：把 Prompt 拆成多个 section，每轮只改 1-2 个，避免一次改太多无法定位问题。

### DSPy 编译器范式

DSPy 提供了一种完全不同的思路——不手写 Prompt，而是声明输入/输出的 Schema（Signature），由编译器自动生成并优化 Prompt。只需定义输入输出字段和验证函数，框架会自动构建少样本示例、搜索最优指令组合。优势在于换模型只需改一行配置，编译器会自动适配。

### 迭代优化真实案例

一个典型的情感分析 Prompt 从基线 70+% 的准确率，经过 2-3 轮 Loop Engineering 迭代（加规则、加反例、调格式），可以提升到 95% 以上。关键洞察是：每次循环只解决 1-2 种错误模式，不是 Prompt 越长越好，而是错误覆盖越全越好。

## 五、Function Calling 进阶

### 多函数定义与并行调用

在实际应用中，模型往往需要同时调用多个工具（如查天气 + 查酒店 + 订机票）。现代模型支持在一次响应中返回多个工具调用（parallel_tool_calls），开发者遍历执行后将结果以 tool 角色回传给模型，模型综合所有结果输出最终回答。

### tool_choice 参数详解

- `auto`：模型自主决定是否调用工具，通用场景
- `none`：不调用任何工具，纯文本对话
- `required`：强制调用一个工具，必须使用工具的任务
- `{"type": "function", "function": {"name": "xxx"}}`：强制调用指定工具，固定流程

### PAL（Program-Aided Language Model）

PAL 让模型写代码代替推理，适用于精确计算类任务。与 Function Calling 不同（模型决定调哪个预定义函数），PAL 让模型自己写任意代码来解决问题，灵活性无限但可靠性和安全性需要额外关注。

### 最佳实践

工具描述的清晰度直接影响模型的选择准确性。单一职责（一个工具做一件事）、参数校验（enum/minimum/maximum）、结构化错误返回是工具定义的核心原则。

## 六、Harness 工程（驾驭工程）

### 范式演进

Prompt Engineering（写指令）→ Context Engineering（给信息）→ Harness Engineering（建环境）。Harness 是围绕 AI 模型构建的一整套运行时系统，解决了"怎么控 AI 才稳"的问题。

### 五大核心组件

**执行环境**：Agent 需要工作场所来读写文件、持久化状态。Workspace 提供文件系统和内存级记忆，StateManager 管理状态流转，让无状态的 LLM 具备状态感知能力。

**工具定义**：将业务逻辑封装为 API 调用，通过 ToolRegistry 统一管理。核心原则包括清晰描述（让模型理解何时调用）、参数校验、单一职责、结构化错误返回。采用渐进式披露策略，不把所有工具塞给模型，需要时动态加载。

**控制循环**：Agent 的核心运转逻辑——感知（接收输入）→ 决策（LLM 决定回复/调用工具/结束）→ 行动（执行工具/输出结果），循环直到无工具调用或达最大轮数。

**防护措施**：多层防护架构——权限最小化（Agent 仅拥有最低权限）、输入净化（防止提示词注入）、输出验证（确保输出符合预期 Schema）、速率限制（防止滥用）。LangChain 实测数据表明，优化 Harness 能使编码 Agent 得分从 52.8% 提升到 66.5%。

**可观测性**：没有观测就无法改进。Tracer 记录 Agent 的每一步（思考过程、工具调用、最终结果），并提供关键指标：任务成功率、平均轮数、Token 消耗、工具调用错误率、响应时间。

### 实战意义

Harness Engineering 的核心信条是"人类掌舵，AI 执行"——AI 负责执行，但由人定义安全的运行边界和合理的控制流程。五大组件的关系是：用户输入经 Guardrails 净化 → Control Loop 调用 LLM → LLM 通过 ToolRegistry 使用工具 → 工具在 Workspace 中执行 → Tracer 记录全过程 → 输出再经 Guardrails 验证后返回用户。
