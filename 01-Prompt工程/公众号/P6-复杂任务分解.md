# Prompt工程 | P6：复杂任务分解

## 准备

对于辅助任务，需要先分解再执行，这样AI才可以不遗忘地完整有序执行。让 LLM 将复杂任务分解为子任务，也是 Agent 学习的基础。

## 核心思想

复杂任务 → 拆解为多个子步骤 → 逐步执行 → 汇总结果

## 方法一：Plan-and-Solve Prompt

在一次对话中，让模型先制定计划，再逐步执行：

```
任务：{task}

请先制定一个执行计划，列出需要完成的子步骤。
格式：
计划：
1. ...
2. ...
3. ...

然后按步骤执行，输出每一步的结果。
```

## 方法二：分步 Prompt + 链式调用

先让模型单独完成拆分任务，再让模型逐个执行子任务：

```python
def step_by_step(task: str) -> str:
    # 先拆解
    messages = [
        {"role": "system", "content": "你是一个任务分解专家。将用户的任务拆解为 3-5 个独立的子任务。"},
        {"role": "user", "content": task}
    ]
    resp = client.chat.completions.create(model="deepseek-v4-flash", messages=messages)
    subtasks = resp.choices[0].message.content
    
    # 对每个子任务逐一执行
    messages_2 = [
        {"role": "system", "content": "逐步执行以下子任务，输出每个步骤的结果。"},
        {"role": "user", "content": subtasks}
    ]
    resp2 = client.chat.completions.create(model="deepseek-v4-flash", messages=messages_2)
    return resp2.choices[0].message.content
```

## 方法三：结构化分解（输出 JSON 子任务列表）

让模型按照特点格式拆分任务，形成子任务列表：

```python
task = "为一家新开的咖啡店制定开业营销方案"
messages = [
    {"role": "system", "content": "将用户的任务分解为子任务列表，输出 JSON 格式：{\"subtasks\": [{\"id\": 1, \"name\": \"...\", \"description\": \"...\", \"priority\": \"high/medium/low\"}]}"},
    {"role": "user", "content": task}
]
```

## 方法四：ReAct 基础

ReAct = Reasoning + Acting，让模型边推理边行动。其中的 Reasoning 部分即为 Day1 P4 的 CoT——ReAct 是将 CoT 从"纯思考"扩展为"思考→行动→观察"的交互式循环：

```
思考：我需要了解 X
行动：搜索 X 的相关信息
观察：X 是 ...
思考：基于以上信息，我需要 ...
行动：计算 ...
观察：结果是 ...
思考：可以给出最终答案了
最终答案：...
```

```
你是一个 AI 助手，使用 ReAct 模式回答问题：
1. Thought: 思考当前需要做什么
2. Action: 执行一个操作（搜索 / 计算 / 查询）
3. Observation: 操作结果
4. 重复以上步骤直到可以给出 Final Answer

问题：北京和上海之间的距离是多少？如果坐高铁需要多久？
```

## 方法五：Tree of Thoughts（思维树）

ToT 即探索多条推理路径，像树一样分叉搜索：

```
问题：用 4 条直线一笔连接下图所有 9 个点，不能抬笔。
. . .
. . .
. . .

请用思维树方法解决：
1. 列出至少 3 种不同的解题思路（树的分支）
2. 对每个思路，评估它的可行性
3. 选择最有希望的思路深入推理
4. 如果第一个思路走不通，回溯到其他分支

格式：
分支1：[思路描述] → 可行性评估 → 尝试推理 → 结果
分支2：[思路描述] → 可行性评估 → 尝试推理 → 结果
...
最终选择：分支X → 最终解决方案
```

ToT 适合"答案空间大、路径多"的开放问题。通常走多条路再选最优，准确率更高但 Token 消耗也更大。

## 多智能体视角的任务分解

当任务足够复杂时，可以让**多个子 Agent 各负责一个子任务**，实现上下文隔离和并行执行：

```
用户 → 主 Agent（编排）
         ├── Agent A：市场调研（独立上下文）
         ├── Agent B：竞品分析（独立上下文）
         └── Agent C：撰写方案（读取 A+B 的输出）
```

每个子 Agent 拥有独立的上下文窗口和工具权限，主 Agent 只负责路由和汇总。这种方式的好处：
- 子 Agent 的详细输出不会污染主对话上下文
- 可以并行执行不依赖的子任务
- 不同子 Agent 可以使用不同模型（复杂任务用强模型，简单任务用低成本模型）

## 注意事项

1. **粒度适中**：子任务太大 → 仍难执行；子任务太小 → 步骤过多
2. **依赖关系**：标注哪些子任务有前后依赖，哪些可以并行
3. **验证点**：每个子任务完成后设置验证条件
4. **回溯机制**：某个子任务失败时，能够重新规划
