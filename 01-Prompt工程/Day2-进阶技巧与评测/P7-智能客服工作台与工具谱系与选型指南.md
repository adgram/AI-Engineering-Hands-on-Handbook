### 第一节：上下文工程（P1）

`context_manager.py` 提供 GSSC 风格的对话窗口管理和相关性评分。

```python
# context_manager.py（核心类结构）
from llm_client import LLMClient

client = LLMClient()

class RelevanceScorer:
    """基于关键词重叠的简单相关性评分器"""
    def __init__(self, threshold=0.3): ...
    def score(self, text: str, query: str) -> float: ...
    def filter(self, items: list[dict], query: str, budget: int = None) -> list[dict]: ...

class ContextManager:
    """对话上下文管理器，维护滑动窗口 + KV Cache 摘要"""
    def __init__(self, system_prompt: str, window_size: int = 10): ...
    def add_message(self, role: str, content: str): ...
    def build_context(self, query: str, include_history: bool = True) -> list[dict]: ...
    def get_kv_cache_summary(self) -> dict: ...
```

#### 设计要点

| 机制 | 说明 |
|------|------|
| 滑动窗口 | 只保留最近 `window_size` 条消息，避免上下文膨胀 |
| 相关性过滤 | 基于关键词重叠过滤历史，保留与当前查询相关的内容 |
| KV Cache 摘要 | `get_kv_cache_summary()` 返回窗口统计和最近主题 |

### 第三节：自动优化（P3）

`prompt_optimizer.py` 内置 Loop Engine 迭代优化客服回复模板。

```python
# prompt_optimizer.py（核心流程）
class PromptOptimizer:
    """Loop Engine：评估 → 收集失败 → 生成变体 → 棘轮保留"""

    def __init__(self, config: OptimizerConfig): ...

    def optimize(self):
        """多轮优化循环"""
        for round_idx in range(max_rounds):
            score = self.evaluate(template)
            failures = self._collect_failures(template)
            error_report = self._analyze_failures(failures)
            new_sections = self._generate_variant(error_report)
            # 棘轮机制：只保留分数不降的版本
            if new_score >= score:
                self.best_template = new_sections

    def evaluate(self, template_text: str) -> float: ...
    def _collect_failures(self, template_text) -> list[dict]: ...
    def _analyze_failures(self, failures) -> str: ...
    def _generate_variant(self, error_report) -> dict: ...
```

#### 棘轮机制

| 组件 | 职责 |
|------|------|
| Evaluator | 在测试集上评估当前模板准确率 |
| Collector | 收集失败案例（expected 不在 output 中的用例） |
| Analyzer | 调用 LLM 分析失败模式，生成错误报告 |
| Generator | 基于错误报告生成改进版模板 |
| Ratchet | 只接受分数 ≥ 当前最优的变体 |

### 第四节：安全防护（P4 + P6）

`guardrails.py` 提供输入过滤和输出验证双重防护。

```python
# guardrails.py（核心类）
class InputSanitizer:
    """输入过滤：检测并过滤 Prompt 注入攻击"""
    def check(self, text: str) -> tuple[bool, str]: ...
    def sanitize(self, text: str) -> str: ...

class OutputValidator:
    """输出验证：检查敏感信息和长度"""
    def check_sensitive(self, text: str) -> bool: ...
    def check_length(self, text: str) -> bool: ...

class DefensePipeline:
    """组合防护流水线"""
    def process_input(self, user_input: str) -> str: ...
    def verify_output(self, output: str) -> str: ...
    def get_stats(self) -> dict: ...
```

#### 防护规则

| 阶段 | 检测项 | 处理方式 |
|------|--------|----------|
| 输入 | 忽略指令、系统提示词攻击、DAN | 过滤替换 + 拦截提示 |
| 输出 | API Key、密码、敏感信息 | 替换为 `[安全拦截]` |
| 输出 | 超长输出（>10000 字符） | 截断 + 提示 |

### 第五节：Function Calling 进阶（P5）

`tools.py` 定义客服系统可调用的工具集。

```python
# tools.py 支持的客服工具
def search_orders(customer_id, status, date_range) -> str: ...
def track_delivery(order_id) -> str: ...
def cancel_order(order_id, reason) -> str: ...
def check_return_policy(product_category) -> str: ...
```

#### 工具 Schema 定义

每个工具遵循 OpenAI Function Calling 格式，包含 `name`、`description`、`parameters` 三个部分。tools.py 中 `TOOL_SCHEMAS` 导出完整 schema 列表，`TOOL_HANDLERS` 导出名称到处理函数的映射。

#### 并行调用能力

模型在一次回复中可以同时调用多个工具（如"查订单 + 查物流"），由后续的 ControlLoop 依次执行。

### 第六节：完整 Harness（P6）

`harness.py` 编排整个工具调用循环。

```python
# harness.py（核心类）
class ToolRegistry:
    """工具注册中心"""
    def register(self, name, schema, handler): ...
    def get_schemas(self) -> list[dict]: ...
    def execute(self, name, **kwargs) -> str: ...
    def get_stats(self) -> dict: ...

class ControlLoop:
    """工具调用主循环"""
    def run(self, messages: list[dict]) -> str: ...

class Tracer:
    """执行追踪器"""
    def log(self, step_type, content, metadata=None): ...
    def summary(self) -> dict: ...
```

#### 执行流程

```
用户输入
    │
    ▼
DefensePipeline.process_input()     ← 安全过滤
    │
    ▼
ContextManager.build_context()       ← 构建上下文（含历史）
    │
    ▼
TaskPlanner.plan_task()              ← 任务拆解
    │
    ▼
ControlLoop.run()                    ← 工具调用循环
    ├── LLM 选择工具
    ├── ToolRegistry.execute()
    └── 循环直到无工具调用或达最大轮数
    │
    ▼
DefensePipeline.verify_output()      ← 输出验证
    │
    ▼
ContextManager.add_message()         ← 记录到历史
    │
    ▼
Tracer.log()                          ← 追踪记录
```

### 一 工具归属谱系

Day2 的所有工具按**掌握时机**分三类：

#### ① Prompt 阶段必须掌握

| 工具/概念                                | 位置 | 为什么必须现在掌握                                     |
| ---------------------------------------- | ---- | ------------------------------------------------------ |

### 二 场景→工具选择矩阵

| 你遇到的问题            | 推荐工具                  | 在哪学          | 不推荐           | 理由                          |
| ----------------------- | ------------------------- | --------------- | ---------------- | ----------------------------- |
| 模型输出格式不稳定      | JSON Mode + Few-shot 示例 | Day1 P3/P5      | DSPy             | 杀鸡用牛刀                    |
| 长对话模型"失忆"        | GSSC：滑动窗口 / 摘要压缩 | P1              | 向量检索         | 前两种实现简单，覆盖 80% 场景 |
| 不知道 Prompt 好不好    | 5 维度评估 + A/B 测试     | Day1 P7 / P3    | DSPy 编译器      | 先手工评估理解原理            |
| 用户试图绕过限制        | 指令隔离 + 角色锚定       | P4              | 仅用正则过滤     | 容易被绕过                    |
| 需要同时查多个 API      | Parallel Tool Calls       | P5              | 串行逐个调用     | 并行省轮次省时间              |
| 复杂逻辑计算            | PAL（写代码执行）         | P5              | Function Calling | FC 需要预注册函数，PAL 更灵活 |
| 想系统性地降 Token 成本 | KV Cache + 缓存策略       | P1 / RAG D6     | 单纯缩短 Prompt  | 要理解缓存命中原理            |
| 想把 Prompt 工程化      | LLMClient + PromptManager | Day1-P7         | Harness 全套     | 先轻量封装，再逐步加组件      |
| 构建完整 AI 应用        | Harness（全部 5 组件）    | P6 / Agent 阶段 | 只用 Prompt 模板 | 不够，需要运行时环境          |
| 需要从文档中找答案      | RAG（阶段二）             | RAG 全系列      | 全塞进 Prompt    | 窗口有限，检索更精准          |
| 需要自主完成复杂任务    | Agent（阶段三）           | Agent 全系列    | 单次 Prompt      | 需要循环 + 工具 + 记忆        |
