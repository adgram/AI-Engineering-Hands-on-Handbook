# P4：向量索引与检索 — 相似度搜索

## 目标
深入理解向量检索的核心算法，掌握不同相似度度量方式的区别

## 相似度度量方式

### 1. 余弦相似度（Cosine Similarity）

```
cosine_sim(A, B) = A·B / (|A| × |B|)
范围: [-1, 1] → 越大越相似
推荐: 文本语义搜索
```

### 2. 点积相似度（Dot Product）

```
dot(A, B) = Σ(Ai × Bi)
范围: 无限制
推荐: 向量已经归一化时，等于余弦相似度
```

### 3. 欧氏距离（Euclidean Distance）

```
euclidean(A, B) = √Σ(Ai - Bi)²
范围: [0, +∞) → 越小越相似
推荐: 图像/音频特征检索
```

## 检索质量评估

检索质量评估的核心原理：用一组预定义的 `query → 期望结果` 测试用例，量化检索系统返回结果的好坏。

| 指标 | 含义 | 测什么 |
|------|------|--------|
| **Hit Rate（命中率）** | 期望结果是否出现在 Top-K 结果中 | "能不能找得到" |
| **MRR（平均倒数排名）** | 第一个相关结果的排序位置倒数取平均 | "找到的排得够不够靠前" |

简单说：Hit Rate 看召回有没有漏，MRR 看排序好不好。

```python
def evaluate_search(collection, queries_with_expected):
    """评估搜索质量"""
    total = len(queries_with_expected)
    hit_count = 0
    mrr = 0  # Mean Reciprocal Rank
    
    for query, expected_ids in queries_with_expected:
        results = collection.query(
            query_texts=[query],
            n_results=5
        )
        
        retrieved_ids = results['ids'][0]
        
        # Hit Rate: 期望结果是否出现在检索结果中
        hit = any(eid in retrieved_ids for eid in expected_ids)
        if hit:
            hit_count += 1
        
        # MRR: 第一个相关结果的排序倒数
        for rank, rid in enumerate(retrieved_ids, 1):
            if rid in expected_ids:
                mrr += 1.0 / rank
                break
    
    hit_rate = hit_count / total
    mrr = mrr / total
    
    return {"hit_rate": hit_rate, "mrr": mrr}

# 使用示例
test_cases = [
    ("什么是 RAG？", ["doc_1"]),
    ("向量数据库怎么用", ["doc_4", "doc_5"]),
]

metrics = evaluate_search(collection, test_cases)
print(f"Hit Rate: {metrics['hit_rate']:.0%}")
print(f"MRR: {metrics['mrr']:.3f}")
```

## 近似最近邻搜索（ANN）原理

向量数据库不可能真的和每个向量逐一计算（太慢），而是使用 ANN 算法：

```
暴力搜索（Brute Force）：O(n)  → 100万文档要算100万次
ANN 搜索：O(log n)  → 使用索引结构加速

常见 ANN 算法：
- HNSW（Hierarchical Navigable Small World）← ChromaDB 默认
- IVF（Inverted File Index）
- PQ（Product Quantization）
```

## ChromaDB 组织多知识库的策略

ChromaDB 通过两层结构组织多个知识库：

```
PersistentClient("chroma_db")     ← 一个数据库目录
├── Collection("kb_excerpts")     ← 名言金句库（集合）
├── Collection("kb_arch_std")     ← 建筑设计规范库（集合）
└── Collection("kb_enum_code")    ← Python 代码库（集合）
```

**什么时候用独立 DB 目录，什么时候用独立集合？**

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 不同项目/租户 | 独立 DB 目录 | 完全隔离，可单独迁移/备份 |
| 同一项目不同领域 | 同一 DB 下不同集合 | 共用 Embedding 模型，统一管理 |
| 测试/生产环境 | 独立 DB 目录 | 环境隔离，避免数据污染 |

代码中通过 `PersistentClient` 的 `path` 参数切换数据库目录，通过 `create_collection` / `get_collection` 的 `name` 参数切换集合。

## 多文件合并知识库

实际 RAG 的知识库通常来自多个文件（JSON、Markdown、Python 代码等）。需要将不同来源的文件合并到同一个 ChromaDB 集合中，并用 `metadata` 标记来源。

```python
# 从三个文件加载
all_docs, all_metas = [], []

# 1) excerpts.json — 每条独立条目
for e in excerpts:
    all_docs.append(e["content"])
    all_metas.append({"source": "excerpts", "file": "excerpts.json"})

# 2) .md 文件 — 按标题分块
md_chunks = re.split(r'\n(?=## )', md_text)
for chunk in md_chunks:
    all_docs.append(chunk)
    all_metas.append({"source": "arch_std", "file": "民用建筑设计统一标准.md"})

# 3) .py 文件 — 按 class/def 分块
py_chunks = re.split(r'\n(?=(class |def ))', py_text)
for chunk in py_chunks:
    all_docs.append(chunk)
    all_metas.append({"source": "enum_code", "file": "enum.py"})

# 统一索引到同一集合
col = client.create_collection("all_sources")
col.add(documents=all_docs, metadatas=all_metas, ids=[...])

# 查询时按 source 过滤
col.query(query_texts=["建筑设计"], where={"source": "arch_std"})
```

ChromaDB 的 `PersistentClient` 可以创建多个数据库目录。每个目录下可以有多个集合，适合按项目或用户隔离知识库：

```
chroma_db_excerpts/         ← 名言金句库
  ├── excerpts              ← 集合1: 所有金句
  └── all_sources           ← 集合2: 多文件合并

chroma_db_arch/             ← 建筑设计知识库
  └── design_standards      ← 集合: 设计规范
```
