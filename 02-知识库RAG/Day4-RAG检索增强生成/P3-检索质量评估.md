## 为什么需要评估？

没有量化评估，优化就失去了方向和依据。通过建立评估体系，可以把每一次参数调整的效果量化，让改进有据可循。

```
没有评估 → 凭感觉调参数 → 越调越糟
有评估 → 数据说话 → 知道每次改动是变好还是变坏
```

## 核心评估指标

评估检索质量需要从不同维度衡量。下面介绍三个核心指标：Hit Rate 衡量是否命中目标文档，MRR 关注第一个正确答案的排名位置，NDCG 则考虑多级相关性排序的完整质量。

### 1. Hit Rate（命中率）

检索结果中是否包含期望的文档。

```
对每个查询：期望文档在 Top-k 结果中 → 算命中
Hit Rate = 命中查询数 / 总查询数
```

### 2. MRR（Mean Reciprocal Rank）

期望文档在结果中的排名。

```
对每个查询：期望文档排名为 r → 贡献 1/r
MRR = 平均(1/r₁ + 1/r₂ + ...)
示例：文档排第1 → 得分1.0；排第3 → 得分0.33
```

### 3. NDCG（Normalized Discounted Cumulative Gain）

考虑结果的多级相关性（不仅仅是相关/不相关）。

```
DCG = Σ(2^rel_i - 1) / log₂(i+1)
NDCG = DCG / IDCG（理想 DCG）
取值 [0, 1]，越高越好
```

## 实现评估工具

下面将零散的评估函数封装为 `SearchEvaluator` 类，统一管理 Hit Rate、MRR 和 NDCG 的计算逻辑，并支持多 k 值批量评估。

```python
import numpy as np
from common.rag_base import BaseRAG

class SearchEvaluator:
    def __init__(self, store):
        self.store = store

    def hit_rate(self, queries_expected: list, k: int = 5) -> float:
        hits = 0
        for query, expected_ids in queries_expected:
            results = self.store.search(query, n_results=k)
            retrieved = results['ids'][0]
            if any(eid in retrieved for eid in expected_ids):
                hits += 1
        return hits / len(queries_expected)

    def mrr(self, queries_expected: list, k: int = 5) -> float:
        reciprocal_ranks = []
        for query, expected_ids in queries_expected:
            results = self.store.search(query, n_results=k)
            retrieved = results['ids'][0]
            found = False
            for rank, rid in enumerate(retrieved, 1):
                if rid in expected_ids:
                    reciprocal_ranks.append(1.0 / rank)
                    found = True
                    break
            if not found:
                reciprocal_ranks.append(0.0)
        return float(np.mean(reciprocal_ranks))

    def ndcg(self, queries_with_grades: list, k: int = 5) -> float:
        scores = []
        for query, id2grade in queries_with_grades:
            results = self.store.search(query, n_results=k)
            retrieved = results['ids'][0]

            dcg = 0
            for i, rid in enumerate(retrieved, 1):
                grade = id2grade.get(rid, 0)
                dcg += (2**grade - 1) / np.log2(i + 1)

            ideal_grades = sorted(id2grade.values(), reverse=True)[:k]
            idcg = sum((2**g - 1) / np.log2(i + 1) for i, g in enumerate(ideal_grades, 1))

            scores.append(dcg / idcg if idcg > 0 else 0)

        return float(np.mean(scores))

    def full_report(self, test_data: dict) -> dict:
        report = {}
        for k in [1, 3, 5, 10]:
            report[f"k={k}"] = {
                "hit_rate": float(self.hit_rate(test_data["binary"], k)),
                "mrr": float(self.mrr(test_data["binary"], k)),
            }
        return report
```

## 实验：找出最优参数

通过网格搜索对 chunk_size、overlap 和 k 值等超参数进行组合遍历，用数据找出检索效果最优的参数配置。

```python
from itertools import product

def find_best_params(store, test_data):
    best_score = 0
    best_params = None

    for chunk_size, overlap, k in product([200, 500, 1000], [0, 50, 100], [3, 5, 10]):
        evaluator = SearchEvaluator(store)
        hr = evaluator.hit_rate(test_data, k=k)
        if hr > best_score:
            best_score = hr
            best_params = (chunk_size, overlap, k)

    return {"best_params": best_params, "best_score": best_score}
```

## 评估最佳实践

建立评估体系并持续优化时，应遵循以下最佳实践，以保障评估的有效性和可靠性。

1. **构建黄金测试集**：人工标注 50-100 条查询 + 期望结果
2. **多维度评估**：不能只看 Hit Rate，也要看 MRR（排序质量）
3. **分层评估**：按文档类型、问题难度分别评估
4. **基线对比**：每次改动都和上一次对比
5. **端到端评估**：除了检索质量，也要评估最终生成质量

> 评估体系说明：本文聚焦检索质量（量化指标），Prompt 篇关注指令清晰度（人工诊断），Agent 篇关注任务完成度（成功率/Trace），三者评估对象不同，可互补参考 → Prompt D1-P7, Agent D9-P6

## 工业案例：检索优化实录

以下三个真实生产调优案例，展示了评估驱动的优化路径：

| 案例 | 场景 | 问题 | 指标 | 优化手段 | 效果 |
|------|------|------|------|---------|------|
| 飞书知识库 | 文档检索 | Recall@10=82%（目标90%） | 召回率 | 动态分块：平均512→320token | 召回率82%→91%，CTR↑18% |
| 抖音客服 | 实时问答 | 响应延迟65ms（目标50ms） | 延迟 | efSearch=100→50，开启ByteCache缓存 | 延迟65ms→38ms，QPS→12000 |
| 金融研报 | 行业检索 | Precision@10=78%（目标85%） | 精确率 | 扩充同义词词典+同义词映射 | 精确率78%→87%，满意度3.8→4.5 |

```python
class CaseStudyAnalyzer:
    """基于字节跳动案例的检索优化分析器"""

    @staticmethod
    def analyze_recall_issue(recall: float, target: float):
        issues = []
        if recall < target:
            if recall < 0.8:
                issues.append("检查分块策略：平均分块长度是否过大？")
                issues.append("建议：按语义单元分块，而非固定token数")
            if recall < 0.85:
                issues.append("检查嵌入模型是否适配领域？")
                issues.append("建议：用领域数据微调嵌入模型")
        return issues

    @staticmethod
    def analyze_latency_issue(latency_ms: float, target_ms: float):
        issues = []
        if latency_ms > target_ms:
            if latency_ms > target_ms * 1.5:
                issues.append("检查HNSW索引efSearch参数（过大会增加耗时）")
                issues.append("建议：开启高频查询缓存，减少重复检索")
            else:
                issues.append("检查负载均衡是否合理")
                issues.append("建议：增加从节点或使用缓存")
        return issues

# 使用
issues = CaseStudyAnalyzer.analyze_recall_issue(0.82, 0.90)
for i in issues:
    print(f"- {i}")
```
