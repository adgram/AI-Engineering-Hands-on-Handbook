# 知识库RAG | P10：拓展阅读 — RAG 进阶知识与前沿技术

本节介绍一些高级、工业化和前沿的 RAG 内容，作为拓展了解。

## 一、缓存策略 — 降低成本提升速度

通过三级缓存避免重复调用 API，降低成本、提升响应速度。P9 的企业级案例就用到了这套机制。

### 三级缓存

```
L1 精确缓存（diskcache）→ L2 语义缓存（ChromaDB）→ L3 调用 API → 结果回写 L1+L2
```

**Embedding 缓存**——用 `md5(model:text)` 为 key 存盘，避免重复向量化。文档入库时尤其有效。

**LLM 精确响应缓存**——以 `sha256(messages+model+temperature)` 为 key，完全一致请求直接返回。适用于 temperature=0。

**语义缓存**——基于向量库，查询相似度低于阈值（默认 0.3）即命中。"RAG 是什么"和"什么是 RAG"能命中同一缓存，这是最有价值的缓存。

### 失效策略

| 策略 | 适用场景 | 特点 |
|------|---------|------|
| TTL | 数据频繁更新 | 定时过期 |
| LRU | 空间有限 | 淘汰最近最少使用 |
| 主动失效 | 数据可追踪 | 文档更新时清除相关缓存 |
| 版本控制 | 系统升级 | 升级后整体失效 |

## 二、成本优化 — Token 用量监控

### 成本公式

```
总费用 = Prompt Tokens × 输入价格 + Completion Tokens × 输出价格
（DeepSeek 输入 ¥0.14/1M、输出 ¥0.28/1M）
```

### 四大优化策略

- **精简 Prompt**——去除冗余修饰词，System Prompt 越短越好
- **压缩历史**——LLM 将多轮对话压缩为摘要
- **分级模型选择**——简单任务用小模型，复杂任务用大模型
- **限制输出长度**——设置 `max_tokens` 避免冗长回答

### 工业实践（字节跳动）

成本构成：GPU 推理 60%、向量存储 25%、网络 15%。优化手段：ByteInfer 量化方案 FP16 显存省 50%、INT8（QAT）省 75%；三级资源调度（高优专属/中优共享/低优夜间空闲资源成本降 70%）。

## 三、高级 RAG 模式 — Self-RAG / Corrective RAG / Adaptive RAG

不同问题适合不同 RAG 策略，三种高级模式让 RAG 更智能地决策何时检索、如何检索。

### 模式对比

| 模式 | 核心思想 | 解决问题 |
|------|---------|---------|
| Naive RAG | 检索→增强→生成 | 基础问答 |
| **Self-RAG** | 生成时自我反思 | 检索时机决策 |
| **Corrective RAG** | 检索后评估质量，不合格重试 | 检索质量保障 |
| **Adaptive RAG** | 动态选择检索策略 | 策略自适应 |

**Self-RAG**：`should_retrieve()` 判断是否需要检索（简单问题不检索省成本），`is_relevant()` 过滤不相关文档。适合需要高准确率的场景。

**Corrective RAG（CRAG）**：检索后评估质量，输出 `relevance/sufficiency/should_retry/rewrite_suggestion` 的 JSON，不合格则重写查询重试（默认 max_retries=2）。适合知识库质量参差不齐的场景。

**Adaptive RAG**：`classify_question()` 将问题分为 factual/reasoning/creative/simple，分别采用不同检索+生成策略。适合混合类型查询。

### 选型指南

| 场景 | 推荐模式 |
|------|---------|
| 知识库质量高 | Naive RAG |
| 知识库质量参差 | Corrective RAG |
| 混合类型查询 | Adaptive RAG |
| 需要高准确率 | Self-RAG |

## 四、带评估的 RAG Pipeline — 实验驱动优化

把 Day4 的检索策略、Rerank、质量评估整合为一个可批量实验、可网格搜索的 Pipeline，形成"策略选择→评估→优化"的闭环。P9 的企业级案例是"产品"，而这是一个"实验工具"。

### 核心方法

`ExperimentRunner` 类串联 retrieve→rerank→generate→evaluate 全流程，`grid_search` 遍历多种配置组合（检索策略 × k 值 × Rerank 开关），输出 avg_faithfulness、avg_relevance 等指标，自动找出最优配置。

## 五、多模态 RAG

### 为什么需要

真实知识库不只有文字——还有图片、图表、表格、扫描件。多模态 RAG 把 Embedding 扩展到图文联合空间。

### 四种方案

**文字描述索引**——用 LLM 生成图片的文字描述存入文本向量库。成本最低，快速原型首选。

**多模态 Embedding**——用 CLIP 将图文映射到同一向量空间，搜文字能找到相关图片。

**统一语义空间**——用 Qwen3-VL-embedding 等模型实现图文互检，适合电商商品搜索。

**Colpali 视觉文档检索**——直接对文档页面截图做视觉编码，保留版面信息。适合 PDF/发票/合同等复杂排版，无需先 OCR。

## 六、分层 RAG — 多级知识库路由

### 为什么需要

单一大库的痛点：噪声多（不相关领域互相干扰）、延迟高（库越大检索越慢）。分层 RAG 按知识层级分发检索，像图书馆按楼层分类。

### 三层 + 回退架构

```
┌─ 全局层（Global）─┐  高频热门知识，小索引，低延迟
├─ 领域层（Domain）─┤  按业务领域划分（finance/tech...），中粒度
├─ 文档层（Document）┤  完整文档库，大索引，高延迟
└─ ask_llm（回退）──┘  简单问候等，LLM 直接回答
```

`HierarchicalRAG` 类用 LLM 做两级路由（先判层级，domain 再判具体领域），`temperature=0.0` 保证路由稳定。

## 七、Agentic RAG — 让 RAG 自主决策

### 传统 RAG vs Agentic RAG

传统 RAG 固定检索 1 次、用原始查询、无评估无追问。**Agentic RAG（A-RAG）** 将 ReAct 循环应用于 RAG，让 LLM 自主决策何时检索、搜什么、是否追问。

| 维度 | 传统 RAG | Agentic RAG |
|------|---------|-------------|
| 检索次数 | 固定 1 次 | 动态多轮 |
| 查询 | 原始查询 | 可改写拆解 |
| 评估 | 无 | 自我反思 |
| 追问 | 无 | 可向用户澄清 |

### 核心循环

```
分析意图 → 决策行动（search/ask/answer）→ 观察结果 → 重复 → 最终回答
```

进阶的 Reflective Agentic RAG 增加自我反思——每次检索后评估信息是否充足，充足则提前终止。

## 八、主流 RAG 框架对比与前沿

### 三大主流框架

**LangChain**：通用 LLM 编排，生态最大，链式组合。适合快速原型和 Agent 场景。

**LlamaIndex**：专注数据框架，索引能力最强，开箱即用。适合复杂数据源和初学者。

**Haystack**：生产级 NLP，管道化设计，部署友好。适合搜索场景和企业级生产。

| 需求 | 推荐框架 |
|------|---------|
| 初学者/快速上手 | LlamaIndex |
| 需要 Agent 能力 | LangChain + LangGraph |
| 搜索场景/生产部署 | Haystack |
| 高度自定义 | 手写 + ChromaDB（本系列做法） |

### RAG 前沿方向

1. **检索即生成的端到端化**：REALM/RETRO/Atlas 把检索内置到训练中
2. **长上下文 RAG**：128K 窗口仍有 Lost in the Middle，需 RAG + Long Context 结合
3. **评估体系化**：Recall/MRR/NDCG/Faithfulness 标准化
4. **多模态 + 工具增强**：图文音统一检索
5. **幻觉检测**：SelfCheckGPT、RAGAS Faithfulness、CRAG

### 工程化差距

当前实践与理想目标的差距：检索准召率 70-85%（目标 95%+）、延迟 1-5s（目标 <500ms）、幻觉率 5-15%（目标 <1%）。

## 九、GraphRAG — 知识图谱增强

### 传统 RAG 的短板

向量检索是"扁平"的——文档间相互独立，无法回答需要跨文档多跳推理的问题（如"A 公司 CEO 毕业于哪所大学"需多次检索链式跳转）。

### GraphRAG 的思路

用知识图谱连接实体，支持多跳推理：

```
传统 RAG：文档1 ←检索→ 文档2 ←检索→ 文档3（多次独立检索）
GraphRAG：  实体A —关系— 实体B —关系— 实体C（图遍历一次到达）
```

### 两种实现

**微软官方 GraphRAG**：文档切分 → 实体提取 → 关系提取 → Leiden 算法实体聚类 → 社区摘要生成 → 查询找相关社区汇总回答。社区摘要让 GraphRAG 能回答"全局性"问题（如"这个领域的主要趋势"）。

**自建轻量知识图谱**：LLM 提取实体 → 同文档实体互联 → 建立实体→文档映射 → 查询时图扩展找相关实体。

### 三方案能力对比

| 方案 | 单文档问答 | 多跳推理 | 全局汇总 | 复杂度 |
|------|-----------|---------|---------|--------|
| 传统 RAG | 强 | 弱 | 弱 | ⭐ |
| GraphRAG（微软） | 中 | 强 | 强 | ⭐⭐⭐⭐ |
| Simple GraphRAG | 中 | 中 | 弱 | ⭐⭐ |
